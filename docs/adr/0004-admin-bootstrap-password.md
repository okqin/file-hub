# 管理员密码首启生成、日志现一次、未改密前每次启动重打

## 决策

管理员用户名固定为 `admin`,不可改。首次启动时,若库中无 admin 行,用 CSPRNG 生成初始密码,以 argon2id 哈希落库(明文绝不入库),并在日志中以 WARN 级别打印**一次**明文。**在管理员尚未自助改密之前,每次进程启动都重新打印当前(或重新生成的)引导密码**;管理员通过控制台自助改密后,停止打印。不提供 CLI 重置入口。

## 背景

PRD 原文设想"管理员凭证由部署配置提供"。本决策将其修订为运行期引导模式:避免在配置/环境变量里管理明文或哈希,降低凭证泄漏面。难点在于"只打印一次"会留下运维缺口——若运维错过那一次日志,单一管理员将被锁死(控制台没有 admin 自助重置入口)。

## Considered Options

- **部署配置提供凭证(PRD 原案)**:简单,但要求运维在 env/secret manager 妥善管理 admin 凭证,与"零配置轻量部署"目标有摩擦。
- **生成一次 + 仅打印一次**:最贴近常规 secret 卫生,但运维错过日志即锁死,无恢复路径。
- **未改密前每次启动重打(选定)**:牺牲"secret 只现一次"的常规直觉,换取"运维总能从最近一次启动日志拿到引导密码"的可恢复性。一旦管理员改密,引导密码失效且停止打印,回归正常 secret 卫生。

## Consequences

- **反直觉**:把明文 secret 重复打进日志,通常是安全反模式;此处是受控的引导例外,且仅在"从未改密"窗口内成立。日志应受部署环境的访问控制保护。
- 管理员自助改密走与普通用户一致的 axum-login `auth_hash` 路径,改密后旧会话自动失效。
- **残留锁死风险**:管理员改密后遗忘密码,将无恢复路径(无 CLI、控制台无 admin 自助重置)。此风险已被显式接受。
